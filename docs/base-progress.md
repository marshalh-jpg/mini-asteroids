# Base step — build progress log

The base build loop (`scripts/ralph-loop.sh`) appends one line here per
completed task in `docs/base-plan.md`.

- Server skeleton and protocol: added `src/game.js` (`Game` class holding ship/asteroids/bullets/score/lives/gameOver), wired a 30 Hz `setInterval` tick loop into `server.js` with `input`/`state` WebSocket message types, broadcasting one state message per tick.
- Ship physics and input: `src/game.js` `tick()` now applies rotate-left/right, thrust along facing angle, per-tick drag, and screen wrap to the ship each tick; fire-key edge detection (from the earlier skeleton) is consumed but bullet spawning is left to a later task.
- Asteroids: `src/game.js` adds `spawnAsteroidWave()` (fixed `ASTEROID_COUNT` large asteroids at game start, spawned away from screen center), straight-line movement with screen wrap in `tick()`, and `splitAsteroid()` (large asteroid -> 2 small asteroids, small asteroid -> destroyed) for the bullet-collision task to call.
- Bullets, collisions, scoring, lives: `src/game.js` `tick()` now spawns a bullet from the ship's nose on fire-edge (`fireBullet()`), moves and expires bullets by fixed lifetime, detects bullet-asteroid and ship-asteroid circle collisions, awards score via `splitAsteroid()`'s return value, and calls `handleShipHit()` to remove a life, respawn the ship centered with a fixed invulnerability window, or set `gameOver` at 0 lives; an emptied asteroid field respawns a fresh wave.
- Client rendering and input capture: rewrote `public/game.js` to track keydown/keyup for left/right/thrust/fire, send an `input` message to the server only when the input state changes, and render the ship, asteroids, bullets, score, lives, and game-over text from each `state` message; no physics or collision logic added client-side.
